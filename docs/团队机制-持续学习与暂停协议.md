# 团队机制：持续学习与暂停协议

目标：让团队持续进化（学习→沉淀→复用），并且在你需要时能“一键停下”切换到交付模式。

## 1. 工作模式（两种模式切换）

### 1.1 Learning Mode（默认）

- 目的：扩充能力、沉淀资产、为下一次迭代做准备。
- 产出形态：笔记、对标分析、demo、清单、设计草图、风险清单。
- 节奏：碎片化推进，不抢你主线的注意力。

### 1.2 Delivery Mode（你发起）

- 触发：你明确说“开始做 X / 进入迭代 / 开发 / 上线 / 本周交付”。
- 目的：围绕单一目标交付可验收成果。
- 约束：严格按角色边界与文档契约推进，减少返工。

## 2. 一键暂停协议（你需要他们停下的时候）

### 2.1 你的指令（任意一句都算）

- “暂停学习，进入交付模式：<目标>”
- “先别学了，帮我把 <任务> 做完”
- “停一下，今天只做 <任务>”

### 2.2 团队暂停后要做的 3 件事（最多 15 分钟完成）

1. 把当前学习进度写到一个统一位置（1 段话即可）：
   - 在对应学习文档里追加 `## 暂停点`：我学到哪、下一步是什么、有哪些链接。
2. 把可复用结论提炼成 3 条（如果来不及，写 1 条也行）：
   - `可复用结论` / `适用条件` / `不适用条件`。
3. 清空 WIP（Work In Progress）：
   - 未完成的学习任务移动到“待续”，不再继续扩展新分支。

### 2.3 暂停状态下的沟通格式

- 只回答：
  - “我当前在做什么”
  - “我能在多快时间交付什么”
  - “需要你做哪一个决策”

## 3. 学习如何“进化角色能力”

### 3.1 每个角色每周固定产出（小而稳定）

- PM：1 页“需求澄清/竞品/范围边界”
- Chief TL：1 张架构草图（Mermaid）或 1 页风险清单
- FE TL：1 页低层设计要点（页面/状态/测试计划）
- BE TL：1 页低层设计要点（接口/数据/测试计划）
- FE Dev：1 个最小 demo 或 1 个可复用组件/脚手架
- BE Dev：1 个最小 demo 或 1 个可复用模块/脚手架
- QA：1 份验收清单或 1 份回归用例模板

### 3.2 能力沉淀的“资产化规则”

- 所有学习结论必须落到仓库文档里（可链接外部资料，但结论要在文档内）。
- 任何重复出现 2 次的问题，必须沉淀成清单/模板/脚手架。
- 任何“做完才能记住”的内容，必须做一个最小可运行 demo。

## 4. 学习 Backlog（避免瞎学）

### 4.1 学习 Backlog 的来源

- 当前迭代遇到的真实问题（优先级最高）。
- 你明确指定的材料（例如 claw0 / deer-flow）。
- 复盘里暴露的短板（例如：部署、SEO、演示、交付）。

### 4.2 学习 Backlog 的每条任务必须包含

- 为什么学（解决什么问题）
- 预计 30/60/120 分钟能产出什么
- 可验收产物是什么（文档/截图/代码/清单）

## 5. 迭代复盘与角色升级

每次迭代结束，做一次 15 分钟复盘，并更新角色能力：

- 本迭代最有效的 3 个动作
- 最浪费时间的 3 个动作
- 下次开始前要新增/删除的角色职责 1-3 条

落地方式：直接修改对应角色的 guideline 文档，并在文件末尾追加 `## Changelog`。
